# Alt3rSnap v0.2 — Design Spec

**Date:** 2026-04-21
**Revision:** 1
**Status:** Draft — awaiting author review
**Scope:** Post-v0.1.1 feature release. Adds edge snap (with ghost preview), middle-click maximize/restore, the full `[[rules]]` TOML schema, and two additional center-sector resize modes (`bottom_right`, `move`). Extends the v0.1 design spec (`docs/superpowers/specs/2026-04-20-alt3rsnap-design.md`, §8.2) without altering its architectural split between the pure-Rust engine and the Win32 adapter.

---

## 0. What's in v0.2

| Feature | User-visible | Implementation scope |
|---|---|---|
| **Edge snap** | Alt+Left-drag near a screen edge/corner shows a translucent ghost preview of the target snap zone; release commits. Fixed zone set (halves, quarters, top-maximize, thirds, bottom-maximize) with per-zone toggles in config. Windows-like proximity + preview UX. Move-only — resize drags never snap. Space (the existing `no_snap` modifier) suspends preview during drag. | New engine module `engine::snap`. New adapter module `overlay.rs` (layered top-level popup). New adapter module `monitors.rs` (enumeration + cache). `tool_window.rs` converts from message-only to hidden top-level to receive broadcast messages. |
| **Middle-click maximize/restore** | Alt + middle-click toggles maximize/restore on the window under the cursor. | New `Event::MiddleDown`. New `Action::ToggleMaximize`. Adapter-side middle-up swallow latch. Config-driven dispatch (`[behavior].middle_click_action`) so future remappings (minimize, close, rollup) slot in without re-plumbing. |
| **Full `[[rules]]` TOML** | Users can write richer exclusions using `match_process` / `match_class` / `match_title` with `exact` / `glob` / `regex` patterns, plus trait masks. Existing `[exclude].processes` keeps working and is additive. | Serde surface opens up. Engine's rule evaluation already supports this shape — no engine logic change. |
| **`bottom_right` + `move` center-resize modes** | `center_mode = "bottom_right"` makes the center sector resize-anchor at the bottom-right corner (AltSnap-compat). `center_mode = "move"` makes the center sector of a right-drag move the window instead of resizing. `symmetric` stays default. | `CenterMode` enum gains two variants. Geometry additions. FSM routing change for `move` (enters `Moving` from `RightDown`). |

**Out of v0.2 scope (deferred to later releases):** `closest_edge` center mode; `include_only` / `override` rule actions (parsed-but-dropped); resize-drag snap; user-defined custom snap zone rects; middle-click remapping to minimize/close/rollup; tray menu additions for snap; in-place preview fade animations; settings GUI; MSI installer; code signing; helper-process IPC.

## 1. Architecture additions

The v0.1 engine/adapter split holds. All additions are additive and obey the existing boundary: no `windows` crate types in `src/engine/**`, no `HWND` leaking across the engine boundary.

### 1.1 New engine modules

- **`src/engine/snap.rs`** — zone baking, proximity detection against a `MonitorSnapshot`, asymmetric hysteresis (engage/disengage thresholds), deterministic tie-break priority, per-`MouseMove` decision function. Called only from the `Moving` state. Pure Rust; no adapter imports.

No new top-level engine state. Snap lives inside `Moving` with additional drag-local snap session fields (see §2).

### 1.2 New adapter modules

- **`src/overlay.rs`** — owns a hidden layered top-level popup used as the snap ghost preview. Responds to `ShowSnapPreview { rect }` / `HideSnapPreview` actions. Simple GDI paint through `SetLayeredWindowAttributes(LWA_ALPHA)` — shared alpha between fill and outline, chosen for v0.2 simplicity.
- **`src/monitors.rs`** — `EnumDisplayMonitors` + `GetMonitorInfoW` + `GetDpiForMonitor` producing a `MonitorSnapshot`. Global cache refreshed outside drags on `WM_DISPLAYCHANGE` and `WM_SETTINGCHANGE(SPI_SETWORKAREA)`.

### 1.3 Modified existing modules

- **`src/engine/mod.rs`** — new events (`MiddleDown`, `DragAborted`), new actions (`ShowSnapPreview`, `HideSnapPreview`, `ApplySnapRect`, `ToggleMaximize`), new drag-local state in `Moving`, centralised `exit_moving(reason)` helper that guarantees preview teardown on every exit path.
- **`src/engine/geometry.rs`** — math for `center_mode = "bottom_right"` (anchor `TopLeft`, i.e., BR corner stays fixed); routing for `center_mode = "move"` (center-sector `RightDown` returns `DragMode::Move` instead of `DragMode::Resize`).
- **`src/engine/rules.rs`** — already supports `Pattern::{Exact,Glob,Regex}` and `RuleAction::{Exclude,IncludeOnly,Override}`. No logic change; just becomes reachable from the serde surface.
- **`src/config.rs`** — `[[rules]]` serde shape, new `[snap]` section, new `[behavior].middle_click_action`, tolerant string-at-file-layer parsing for `center_mode` and `middle_click_action` so the bridge can warn and default on unknown values.
- **`src/hook.rs`** — `WH_MOUSE_LL` translates `WM_MBUTTONDOWN` into `Event::MiddleDown`. `WM_MBUTTONUP` is handled entirely at the adapter level via the swallow latch and is **not** surfaced to the engine. `WM_DISPLAYCHANGE` and `WM_SETTINGCHANGE` branches added to the tool window.
- **`src/adapter.rs`** — handlers for the four new actions; resolves the current `MonitorSnapshot` on move-drag `BeginDrag` and attaches it to the engine event; middle-up swallow latch state (`AtomicBool` + `SetTimer`).
- **`src/tool_window.rs`** — converts from message-only (`HWND_MESSAGE` parent) to a hidden **top-level** utility window. Message-only windows do not receive broadcast messages; `WM_DISPLAYCHANGE` and `WM_SETTINGCHANGE` are broadcast-only. Styles: `WS_OVERLAPPED` + `WS_EX_TOOLWINDOW | WS_EX_NOACTIVATE`; hidden for process lifetime. Single window — no separate broadcast helper.

### 1.4 Unchanged modules

`main.rs`, `tray.rs`, `fullscreen.rs`, `dpi.rs`, `autostart.rs`, `elevate.rs`, `config_ops.rs`, `win_api.rs` (gains `is_zoomed` / `show_window` wrappers only — no architecture change).

### 1.5 Threading

Still single-threaded, main-thread-only. All new work (overlay creation, monitor enumeration, cache refresh, latch bookkeeping) runs from hook callbacks and the tool window's message pump. The monitor cache sits behind a `Mutex<MonitorSnapshot>` — read on `BeginDrag`, written on broadcast handlers, never contended on the hot path.

### 1.6 Monitor layout: snapshot at BeginDrag

The engine treats the `MonitorSnapshot` attached at `BeginDrag` as immutable for the lifetime of the drag. A global cache in the adapter is refreshed outside of drags via `WM_DISPLAYCHANGE` + `WM_SETTINGCHANGE(SPI_SETWORKAREA)` on the hidden top-level tool window. If a display layout change arrives mid-drag, the in-flight drag finishes on its old snapshot; the refreshed cache applies to the next drag. There is no engine-visible `MonitorLayoutChanged` event.

```rust
struct MonitorSnapshot {
    monitors: Vec<MonitorInfo>,
}
struct MonitorInfo {
    bounds: Rect,      // full monitor rect (used for edge-proximity detection)
    work_area: Rect,   // taskbar-excluded; snap zones derive from this
    scale: u32,        // DPI numerator (e.g., 150 for 150%); not consumed by v0.2 math
}
```

Zone rects derive from `work_area`. Edge-proximity detection uses `bounds` so that top-edge intent still fires from `y = 0` even when an auto-hidden taskbar makes `work_area.top > bounds.top`.

## 2. Engine shape

### 2.1 New event and action types

```rust
// in src/engine/state.rs

pub enum Event {
    // existing variants unchanged
    MiddleDown { cursor: Point },
    DragAborted { reason: DragAbortReason },
}

pub enum DragAbortReason {
    TargetInvalid,
    CaptureLost,
    ApplyGeometryFailed,
}

pub enum Action {
    // existing variants unchanged
    ShowSnapPreview { rect: Rect },
    HideSnapPreview,
    ApplySnapRect   { hwnd: WindowId, rect: Rect },
    ToggleMaximize  { hwnd: WindowId },
}

pub enum DragOrigin {
    PrimaryButton,      // normal LeftDown → Move, or RightDown → Resize
    CenterMoveMode,     // RightDown in center sector with center_mode = "move"
}

pub enum CenterMode {
    Symmetric,
    BottomRight,
    Move,
    // `ClosestEdge` reserved for v0.3
}
```

`DragAborted` semantics: the current drag cannot safely continue. Reserved for target invalidation, irrecoverable capture loss, and geometry application failures that invalidate drag continuation. Transient `SetWindowPos` failures (e.g., the existing UIPI-denial balloon path) stay on the warn-and-continue channel; they do **not** emit `DragAborted`.

All actions use `WindowId` uniformly. `HWND` appears only in adapter code.

### 2.2 `BeginDrag` extension (move drags)

```rust
Action::BeginDrag {
    hwnd: WindowId,
    initial_rect: Rect,
    grab: Point,
    mode: DragMode,
    snap: Option<SnapContext>,   // Some(..) on move drags when snap is available; None otherwise
}

struct SnapContext {
    monitors: MonitorSnapshot,
    zones: SmallVec<[SnapZone; 11]>,   // baked from config + snapshot at drag start
    engage_px: u32,
    disengage_px: u32,
}
```

Snap zones are materialised as absolute `Rect`s once, at `BeginDrag`. The hot path on `MouseMove` is a bounded linear scan over at most 11 zones with integer arithmetic only.

### 2.3 `Moving` state fields

```rust
struct MovingState {
    hwnd: WindowId,
    initial_rect: Rect,
    grab: Point,
    drag_origin: DragOrigin,
    snap_session: Option<SnapSession>,   // None only when snap feature unavailable for this drag
    pending_passthrough: bool,           // existing
}

struct SnapSession {
    monitors: MonitorSnapshot,
    zones: SmallVec<[SnapZone; 11]>,
    engage_px: u32,
    disengage_px: u32,
    engaged: Option<EngagedZone>,
    last_preview_rect: Option<Rect>,     // adapter-dedup assist
    suspended: bool,                     // true while Space is held; initialised from initial Space state at BeginDrag
}

struct EngagedZone {
    id: SnapZoneId,
    target_rect: Rect,
    entered_at_cursor: Point,
}
```

`snap_session` presence encodes feature availability for the drag. Suspension (Space held) lives entirely inside `SnapSession.suspended`; the option never flips to `None` in response to a modifier change. This lets Space release mid-drag resume snap evaluation cleanly because the session already exists.

### 2.4 Live window motion is always emitted

`snap::evaluate` never suppresses live drag motion. Every `MouseMove` in `Moving` emits `UpdateDrag { unsnapped_rect }` alongside any preview actions. The real window keeps following the cursor; the preview is additive.

| `snap::evaluate` result | Emitted actions |
|---|---|
| `Engage(new_zone)` (no prior engagement) | `ShowSnapPreview { rect: new_zone.target_rect }` + `UpdateDrag { unsnapped_rect }` |
| `Engage(new_zone)` (different from current) | `ShowSnapPreview { rect: new_zone.target_rect }` + `UpdateDrag { unsnapped_rect }` — **no transient `HideSnapPreview`**; the overlay repositions in place |
| `Hold(zone)` | `UpdateDrag { unsnapped_rect }` (no preview action — deduped via `last_preview_rect`) |
| `Disengage` | `HideSnapPreview` + `UpdateDrag { unsnapped_rect }` |
| `None` | `UpdateDrag { unsnapped_rect }` |

### 2.5 Deterministic zone priority

When multiple zones are eligible at a cursor position, priority (highest wins):

```
1. Corner quarters            (top-left, top-right, bottom-left, bottom-right)
2. Top edge → maximize         (matches Aero)
3. Bottom edge → bottom-maximize (mirror; off by default)
4. Halves                      (left-half, right-half)
5. Thirds                      (left-third, middle-third, right-third; off by default)
```

The order is fixed in v0.2 (not user-configurable). Disabling a zone removes it from the candidate set; the next-highest-priority eligible zone wins. This is part of the `engine::snap` contract and is covered by table tests.

### 2.6 Key transitions

- **`Armed + MiddleDown`** on a rule-eligible window with `middle_click_action = ToggleMaximize` → `[ToggleMaximize { hwnd }, SwallowEvent]`. State stays `Armed`. Excluded or `middle_click_action = None` → no actions, no swallow.
- **`Moving + MouseMove`** — runs `snap::evaluate` (see §2.4 table). While `snap_session.suspended`, `evaluate` returns `None` and any prior engagement is torn down at the moment of suspension.
- **`Moving + LeftUp`** with engaged zone → emitted in exact order:
  ```
  1. HideSnapPreview
  2. ApplySnapRect   { hwnd, rect: engaged.target_rect }
  3. EndDrag         { hwnd }
  4. CancelMenuActivation
  ```
- **`Moving + LeftUp`** with no engagement → existing `EndDrag` + `CancelMenuActivation` path (unchanged from v0.1).
- **`Moving + KeyChange(Space, down=true)`** when `ActivationPolicy::no_snap` matches → set `snap_session.suspended = true`, hide any active preview (`HideSnapPreview` + clear `engaged`).
- **`Moving + KeyChange(Space, down=false)`** → clear `suspended`. Next `MouseMove` resumes normal evaluation.
- **`Moving + DragAborted`** → teardown path (see §2.7). No `ApplySnapRect` — aborted drags never commit snap.
- **`Armed/Resizing + RightDown` in center sector** with `center_mode = "move"` → enter `Moving` with `drag_origin = CenterMoveMode`. This is a routing change at the entry point, not a mode transition mid-drag. Because the resulting state is `Moving`, the drag **is eligible for snap**: `BeginDrag` carries `snap: Some(..)` on the same terms as any other move drag. `drag_origin` exists to distinguish the entry path for logs and tests, not to gate snap.

### 2.7 Exit-from-`Moving` invariant

Any transition out of `Moving` must, before the state actually changes, inspect `snap_session.engaged` and emit `HideSnapPreview` if set. Exit paths covered:

- `LeftUp` (normal end — with or without commit)
- `FullscreenFocused` resolved via `pending_passthrough` (preview teardown happens at the natural `EndDrag`)
- `DragAborted { reason }` (target destroyed, capture lost, `ApplySnapRect`/`UpdateDrag` failed non-recoverably)
- `ToggleEnable` → disabled mid-drag (preview hidden, drag ends)
- `ReloadConfig` mid-drag (preview hidden only if `engaged` was already set at `LeftUp`; zones stay baked — see §2.8)

A helper `Engine::exit_moving(reason)` centralises "hide preview if engaged + the reason-specific cleanup" so no exit path can drift.

### 2.8 `ReloadConfig` during an active drag

Drag-local baked zones are immutable until the drag ends, even if the new config disables snap entirely. New config applies to the next drag. If the current drag has `snap_session.engaged = Some(..)` at `LeftUp`, the `ApplySnapRect` commit still fires — snap ignores the mid-drag config change. This avoids half-applied snap behavior.

### 2.9 Middle-click swallow is a click-sequence contract

> **Contract:** if the engine acts on `MiddleDown`, the adapter suppresses the entire middle-click sequence (both down and up) for that button press.

Implementation:
- Engine sees `MiddleDown` only. On action, emits `[ToggleMaximize, SwallowEvent]`.
- Adapter, on processing that batch, sets `swallow_next_middle_up = true` (an `AtomicBool`).
- The mouse hook, on the next `WM_MBUTTONUP`, returns `LRESULT(1)` if the latch is set (and clears it).

Latch cleanup — the primary clear is "matching `WM_MBUTTONUP` arrived." Two defensive secondaries:
- A 500 ms `SetTimer` safety clear (in case the up never arrives).
- Clear on the next `BeginDrag` processed (a new drag should never inherit a stale latch).

These secondaries are defensive cleanup, not a coordinated sub-FSM.

### 2.10 Action idempotence (adapter-side tolerance)

- `ShowSnapPreview { rect }` is idempotent for the same rect (no-op at the overlay).
- `HideSnapPreview` is harmless when the overlay is already hidden.
- Engine dedupes via `last_preview_rect`; the adapter also short-circuits identical calls. Belt-and-suspenders.

### 2.11 `ApplySnapRect` is terminal for drag geometry

> **Contract:** `ApplySnapRect` is the final geometry-setting action for a drag. No action after it in the same batch may call `SetWindowPos`. `EndDrag` is explicitly non-geometric — it performs `ReleaseCapture` and adapter-state cleanup only.

Enforced by a `RecordingWinApi` mock in `tests/adapter_ordering.rs` that panics if any `set_window_rect` call is recorded after `apply_snap_rect` within a single `apply_actions` invocation.

## 3. Win32 adapter additions

### 3.1 `src/overlay.rs` — snap preview overlay (~150 LOC)

A single hidden layered top-level window, class `Alt3rSnapOverlay`, registered once at startup.

- **Styles:** `WS_POPUP` + `WS_EX_LAYERED | WS_EX_TRANSPARENT | WS_EX_NOACTIVATE | WS_EX_TOOLWINDOW | WS_EX_TOPMOST`.
  - `WS_EX_LAYERED` — required for alpha.
  - `WS_EX_TRANSPARENT` — mouse events pass through; the ghost never steals clicks.
  - `WS_EX_NOACTIVATE | WS_EX_TOOLWINDOW` — excluded from Alt-Tab, never focusable.
  - `WS_EX_TOPMOST` — sits above the live-following dragged window.
- **Lifecycle:** lazy-created on first `ShowSnapPreview`, kept alive for process lifetime afterwards. Destroyed only at app exit.
- **Paint model — shared alpha, v0.2 commitment:**
  - `SetLayeredWindowAttributes(hwnd, 0, preview_opacity, LWA_ALPHA)` applied once per show. `preview_opacity` defaults to `0x99` (~60%), sourced from `AdapterConfig`.
  - `WM_PAINT` handler: `FillRect` interior with accent colour; `FrameRect` 2 px border with the same accent colour. Both pixels end up at window-level alpha — there is no per-pixel opacity channel in v0.2.
  - Accent colour: `DwmGetColorizationColor` at startup; fallback `RGB(0, 120, 215)`.
  - `UpdateLayeredWindow` is **not** used in v0.2. The two alpha models are mutually exclusive for a window's lifetime; we pin `SetLayeredWindowAttributes` up front. Switching to `UpdateLayeredWindow` for per-pixel alpha is a tracked post-v0.2 polish item.
- **API (adapter-internal):**
  ```rust
  overlay::show(rect: Rect);   // SetWindowPos + SW_SHOWNA; no-op if already visible at same rect
  overlay::hide();             // SW_HIDE; no-op if already hidden
  ```

### 3.2 `src/monitors.rs` — monitor snapshot + cache (~80 LOC)

```rust
fn snapshot() -> MonitorSnapshot {
    // EnumDisplayMonitors → for each HMONITOR:
    //   GetMonitorInfoW  → bounds (rcMonitor), work_area (rcWork)
    //   GetDpiForMonitor(MDT_EFFECTIVE_DPI) → scale
    // Build Vec<MonitorInfo>.
}
```

Global cache behind `Mutex<MonitorSnapshot>`. Refreshed:
- On first access (lazy init at startup).
- On `WM_DISPLAYCHANGE` received by the hidden top-level tool window.
- On `WM_SETTINGCHANGE` with `wParam == SPI_SETWORKAREA` (filter; this message fires for many categories).

Refresh is debounced at 200 ms — on rapid successive broadcasts only the last refresh runs. Refresh never runs during an active drag; broadcasts that arrive mid-drag update a pending flag that fires when the drag ends.

### 3.3 `src/hook.rs` changes

- `WH_MOUSE_LL` gains a `WM_MBUTTONDOWN` branch: translate to `Event::MiddleDown { cursor }`, feed to engine.
- `WM_MBUTTONUP` handling:
  ```rust
  unsafe fn on_wm_mbuttonup(code: i32, w: WPARAM, l: LPARAM) -> LRESULT {
      if SWALLOW_MIDDLE_UP.swap(false, Ordering::Relaxed) {
          return LRESULT(1);
      }
      CallNextHookEx(None, code, w, l)
  }
  ```
- Horizontal and other mouse buttons unchanged.

### 3.4 `src/tool_window.rs` — hidden top-level

Converts from `HWND_MESSAGE`-parented to a hidden top-level window (see §1.3). Message handlers gain:
- `WM_DISPLAYCHANGE` → post `monitors::request_refresh()`.
- `WM_SETTINGCHANGE` (with `wParam == SPI_SETWORKAREA` filter) → same.
- `WM_TIMER` (new timer id for the 500 ms middle-up safety clear) → clear `SWALLOW_MIDDLE_UP` if set.

Existing handlers (`WM_APP+n` custom traffic, tray callback) unchanged.

### 3.5 `src/adapter.rs` — action handlers

- `ShowSnapPreview { rect }` → `overlay::show(rect)`.
- `HideSnapPreview` → `overlay::hide()`.
- `ApplySnapRect { hwnd, rect }` → `win_api::set_window_rect(hwnd, rect)`. Hard failure → emit `Event::DragAborted { reason: ApplyGeometryFailed }` back into the engine. UIPI-denial path unchanged (rate-limited balloon, log at warn, no abort).
- `ToggleMaximize { hwnd }` → branch on `IsZoomed(hwnd)` → `ShowWindow(hwnd, SW_RESTORE)` or `ShowWindow(hwnd, SW_MAXIMIZE)`. Also sets `SWALLOW_MIDDLE_UP = true`.

Action-batch preprocessing: the adapter inspects each incoming `Vec<Action>` for `BeginDrag` before processing; if present, `SWALLOW_MIDDLE_UP` is cleared (defensive — no stale latch into a new drag).

### 3.6 `src/win_api.rs` additions

Thin wrappers:
- `is_zoomed(hwnd) -> bool`
- `show_window(hwnd, SW_*)`

Nothing existing in `win_api.rs` changes.

### 4.2 File-layer serde shapes

`center_mode` and `middle_click_action` are `String` at the file layer so the bridge can warn and default on unknown values. (Serde-into-Rust-enum would reject the file before the bridge runs.)

```rust
#[derive(Deserialize)]
struct ResizeFile {
    #[serde(default = "default_center_mode_str")]
    center_mode: String,                   // STRING at file layer
    #[serde(default = "default_center_fraction")]
    center_fraction: f32,
}

#[derive(Deserialize)]
struct BehaviorFile {
    // existing fields
    #[serde(default = "default_middle_click_action_str")]
    middle_click_action: String,           // STRING at file layer
}

#[derive(Deserialize)]
#[serde(rename_all = "snake_case")]
enum PatternFile {
    Exact(String),
    Glob(String),
    Regex(String),
}

#[derive(Deserialize)]
#[serde(rename_all = "snake_case")]
enum RuleActionFile {
    Exclude,
    IncludeOnly,                           // parsed for forward-compat; dropped at bridge
    Override(PerWindowConfigFile),         // same
}

#[derive(Deserialize)]
struct RuleFile {
    match_process: Option<PatternFile>,
    match_class:   Option<PatternFile>,
    match_title:   Option<PatternFile>,
    #[serde(default)]
    match_traits:  WindowTraitMaskFile,
    action:        RuleActionFile,
}

#[derive(Deserialize)]
struct SnapFile {
    #[serde(default = "default_true")]           enabled: bool,
    #[serde(default = "default_engage")]         engage_distance_px: u32,
    #[serde(default = "default_disengage")]      disengage_distance_px: u32,
    #[serde(default = "default_preview_opacity")] preview_opacity: u8,
    #[serde(default)]                            zones: ZoneToggles,
}
```

### 4.3 Validation rules (run in `to_runtime_config`)

1. `snap.disengage_distance_px >= snap.engage_distance_px`. Violation → log warn, clamp `disengage` up to equal `engage`.
2. `snap.engage_distance_px <= 256`. Safety cap; violation → log warn, clamp.
3. `resize.center_mode` matched against `{"symmetric", "bottom_right", "move"}`. Unknown → log warn, default to `"symmetric"`. `"closest_edge"` is explicitly reserved; treat as unknown.
4. `behavior.middle_click_action` matched against `{"none", "toggle_maximize"}`. Unknown → log warn, default to `"none"` (fail-safe: do nothing rather than guess).
5. `action = "include_only"` or `action = "override"` on a rule → log warn `"rule action {name} is not supported in v0.2; rule dropped"` and skip the rule.
6. Rule with all predicates absent (`match_process`, `match_class`, `match_title`, `match_traits` all empty) → log warn `"rule #{i} has no predicates — dropping to avoid match-everything footgun"` and skip the rule. No implicit match-all primitive in v0.2.
7. Regex compile errors → log warn with rule index, drop the failing rule, keep others.
8. `[exclude].processes` entries are lowercased and desugared into `WindowRule { match_process: Pattern::Exact(lowercase), action: Exclude }`, then **prepended** to the `[[rules]]` vector. Rules evaluate in config order; the first match wins.

### 4.4 Case-folding semantics

- **Process-name patterns** (`match_process`) are **always case-insensitive**:
  - `Exact("MSTSC.exe")` compiles to `Exact("mstsc.exe")`.
  - `Glob("Game_*.EXE")` compiles to `Glob("game_*.exe")`.
  - `Regex("Chrome\\.exe|Firefox\\.exe")` compiles with `RegexBuilder::case_insensitive(true)`, overriding any inline flags the user wrote — for `match_process` the guarantee is absolute.
  - The engine match path compares lowercase-to-lowercase; no per-match work.
- **Class-name and title patterns** (`match_class`, `match_title`) are **case-sensitive by default**. Users who want case-insensitive class or title matching use `Regex` with inline `(?i)`.

Documented in the user-facing config reference.

### 4.5 Runtime config split

The bridge produces two halves:

```rust
pub struct RuntimeConfig {
    pub engine: EngineConfig,
    pub adapter: AdapterConfig,
}

pub struct EngineConfig {
    pub activation: ActivationPolicy,
    pub behavior:   BehaviorConfig,        // + middle_click_action: MiddleClickAction
    pub resize:     ResizeConfig,          // + center_mode: CenterMode
    pub snap:       SnapEngineConfig,      // zones, engage_px, disengage_px, enabled
    pub rules:      Vec<WindowRule>,
}

pub struct AdapterConfig {
    pub preview_opacity: u8,               // overlay paint — engine never sees this
    // reserved: preview_accent_override, balloon_cooldown_s, etc.
}
```

Overlay-paint knobs live in `AdapterConfig` so the engine boundary stays clean. The engine never reads `preview_opacity`.

### 4.6 Hot reload

Tray "Reload config" hot-swaps both halves: `engine.set_config(runtime.engine)` + `adapter.set_config(runtime.adapter)`. The drag-in-progress invariant from §2.8 applies to the engine half (baked snap zones are immutable). The adapter half (opacity) applies immediately — it only affects future `ShowSnapPreview` paints.

### 4.7 Migration

No migration required for v0.1.1 users. Existing configs load verbatim — `[snap]`, `[[rules]]`, new `[behavior]` / `[resize]` keys all default to safe values. First `[snap]` write-back (if any) contains defensive defaults only.

## 5. Testing plan

All automated tests stay pure-Rust and Linux-runnable. Adapter-level ordering and latch contracts are asserted via a mock `WinApi` trait that keeps the invariants in CI.

### 5.1 New file — `tests/engine_snap.rs`

- **Zone geometry:** halves / quarters / thirds / top-maximize computed against a `MonitorSnapshot` whose `work_area` is strictly smaller than `bounds` (simulates a bottom taskbar). Property (via `proptest`): zone rects never escape `work_area`.
- **Proximity with asymmetric hysteresis:**
  - Cursor approaching the `LeftHalf` trigger edge crosses 24 px → single `Engage(LeftHalf)`.
  - Cursor oscillates around the 24 px boundary → no re-engage until it has left beyond 32 px first.
  - Cursor crosses from `LeftHalf` region directly into `TopLeftQuarter` region → two events: `Disengage` then `Engage(TopLeftQuarter)`.
- **Zone-switch in a single `MouseMove`:** engaged on `LeftHalf`; next `MouseMove` makes `TopLeftQuarter` the winner. Emitted actions are exactly `[ShowSnapPreview { new_rect }, UpdateDrag { unsnapped_rect }]` — **no transient `HideSnapPreview`**. This is the contract that lets the overlay reposition in place.
- **Tie-break priority:** cursor at a point eligible for quarter + half + maximize simultaneously → highest-priority zone wins. Table-driven across every overlap corner case.
- **Space suspension:**
  - Engaged zone + `KeyChange(Space, down=true)` → `HideSnapPreview` emitted + `engaged` cleared + `suspended = true`.
  - During `suspended`, subsequent `MouseMove` emits only `UpdateDrag`.
  - `KeyChange(Space, down=false)` → next `MouseMove` resumes normal evaluation.
  - Space already held at `BeginDrag` → `snap_session.suspended = true` from the start.
- **Zone immutability across `ReloadConfig`:** bake `SnapContext` at `BeginDrag` from snapshot A. Feed `ReloadConfig` with `snap.enabled = false`. Continue drag → zones still evaluated against A; release on engaged zone → still commits.
- **Cross-monitor:** cursor moves from monitor A to monitor B during a single drag. Zones on B are evaluated against B's `work_area` from the same snapshot. Adapter produces a full multi-monitor snapshot; engine never requeries mid-drag.

## 7. Risks, open questions, and pinned decisions

### 7.1 Risks (bounded; mitigations in place)

- **Overlay paint perceived quality.** Shared-alpha via `SetLayeredWindowAttributes` means the outline is not fully opaque over a soft fill. Users comparing side-by-side with Windows' own Snap preview may notice. *Mitigation:* ship v0.2 as-is; tracked post-v0.2 as a polish lever that would switch to `UpdateLayeredWindow`. Not a release blocker.
- **Hysteresis defaults on high-DPI displays.** 24 px / 32 px are physical pixels — on a 4K/200% monitor the engage feel may be tight; on 1080p/100% it may be loose. *Mitigation:* values are in `[snap]` config from day one. If release-gate testing surfaces a systemic feel issue, defaults shift before tag. No architectural recourse needed.
- **Snap preview + maximised-restore interaction.** Alt + Left-drag on a maximised window restores it, then the drag continues. Snap's zone evaluation starts firing from the first `MouseMove` after restore. If the cursor is already near a screen edge immediately post-restore, the preview may engage unexpectedly. *Implementation must choose and document a guard behaviour during M4* (e.g., ignore snap for N ms after `RestoreIfMaximized`). Tracked as an open engine detail for M4; not left for implementers to improvise silently.
- **UIPI + elevated target windows.** v0.1 already handles the unelevated-Alt3rSnap vs. elevated-target case with a rate-limited balloon. Snap adds `ApplySnapRect` as another geometry mutation that can fail under UIPI. *Mitigation:* `ApplySnapRect` runs through the same failure path as `UpdateDrag`; hard failure emits `DragAborted { reason: ApplyGeometryFailed }` (§2.1), preview hides normally, user sees the existing balloon. No new surface.
- **`WM_SETTINGCHANGE(SPI_SETWORKAREA)` noise.** This message fires for many categories; we filter on `wParam == SPI_SETWORKAREA`, but misfiring apps are common. *Mitigation:* the refresh handler is cheap (one `EnumDisplayMonitors` pass), debounced at 200 ms, and never runs mid-drag. Worst case: an extra cache refresh.
- **Pattern-matching regex and untrusted input.** The `regex` crate is safe against ReDoS by design. Rules come from the user's own `config.toml`, not from the network. Not a security risk; noted for completeness.

### 7.2 Open questions to resolve during implementation

1. **Preview fade-in/out?** v0.2 flashes the overlay on/off instantaneously. A short fade (80–120 ms) would feel more polished. *Decision deferred to M4*; default is no fade (simpler, no animation timer).
2. **DPI scaling in snap math.** The engine works in physical pixels throughout; monitors carry `scale` in the snapshot for future use but v0.2 does not consume it. Flagged so M4 implementation does not accidentally introduce DPI scaling into snap math for no reason.
3. **Maximized-restore guard value.** See §7.1 bullet — tie the specific ms value (or the choice of a different guard shape) to M4 implementation, but require it be picked and documented in the M4 plan.

### 7.3 Pinned decisions (not open — recorded so they don't drift)

- **Top-edge vs. halves priority.** Corners > top-maximize > halves is pinned in v0.2. If a user enables `top_maximize = true` + `left_half = true` + `top_left_quarter = false`, far-left portions of the top edge pick top-maximize (not halves). The priority table can open up via config in v0.3 if feedback warrants; v0.2 does not re-open it.
- **Tray menu additions.** None in v0.2. Snap toggles live in config; middle-click remapping lives in config. If users repeatedly ask for in-tray snap on/off, adding a single "Snap" checkmark is a small post-v0.2 change — noted as a clean lever, not a v0.2 item.
- **Snap enabled by default on upgrade.** `snap.enabled = true` with the Aero-like zone subset. v0.1 users upgrading to v0.2 get snap *on* without opting in; preview appears on their next Alt + Left-drag near an edge. This is intended — the feature is the headline.
- **Move-only snap.** Resize drags never preview. (Brainstorming Q8.)
- **Fixed zone set with per-zone toggles.** No user-defined rect zones in v0.2. (Brainstorming Q5.)
- **Middle-click semantics.** Acts on `MiddleDown` while `Armed`; no modifier combos, no middle-drag. (Brainstorming Q6.)
- **`closest_edge` not shipped.** `bottom_right` + `move` only. (Brainstorming Q7.)
- **`[exclude].processes` stays forever-compatible.** Not deprecated. (Brainstorming Q9.)

## 8. Glossary (v0.2 additions)

- **Snap zone** — a named rect derived from a monitor's `work_area` that the snap preview can engage on (e.g., left-half, top-left quarter, top-maximize).
- **Asymmetric hysteresis** — engage and disengage proximity thresholds are different values (24 px / 32 px by default) so that a cursor lingering near the boundary does not flicker in and out of engagement.
- **`SetLayeredWindowAttributes(LWA_ALPHA)`** — the "whole-window alpha" layered-window path. Mutually exclusive with `UpdateLayeredWindow` (per-pixel alpha) for a window's lifetime.
- **`WM_DISPLAYCHANGE`** — broadcast when display resolution or layout changes. Received only by top-level windows.
- **`WM_SETTINGCHANGE(SPI_SETWORKAREA)`** — broadcast when work area changes (taskbar moved/resized, reserved regions updated). Received only by top-level windows.
- **Drag-local snap session** — the `SnapSession` record inside `MovingState`; holds the baked zones, engagement state, suspension flag, and dedup assist for one drag. No global snap state.
