## 2. Lint configuration

### 2.1 rustfmt.toml

New file at repo root. Locks in currently-implicit defaults so `cargo fmt` output is stable across toolchain upgrades.

```toml
edition = "2021"
max_width = 100
imports_granularity = "Crate"
group_imports = "StdExternalCrate"
use_field_init_shorthand = true
use_try_shorthand = true
```

After adding this file, run `cargo fmt --all` once to apply any reformat (expected: minimal churn since these match current style).

### 2.2 [lints] table in Cargo.toml

Rust 1.73+ feature, within the project's `rust-version = "1.75"` MSRV. Moves lint control into the manifest, keeping source files clean of `#![...]` attributes.

```toml
[lints.rust]
unused_import_braces = "warn"
unused_qualifications = "warn"
unused_lifetimes = "warn"

[lints.clippy]
all = "warn"
```

The `clippy::all = "warn"` entry makes the manifest authoritative and matches the current `cargo clippy -- -D warnings` CI invocation. No pedantic lints in this release.

## 3. windows crate feature audit

### 3.1 Findings

Grepping `use windows::` across `src/` shows the following features from `Cargo.toml` have **no live imports**:

| Feature | Status |
|---|---|
| `Win32_System_Com` | No imports found — candidate for removal |
| `Win32_Storage_FileSystem` | No imports found — candidate for removal |
| `Win32_System_SystemInformation` | No imports found — candidate for removal |

All other 12 declared features have confirmed imports.

### 3.2 Approach

Remove the three candidates from `Cargo.toml`. Verify with `cargo build --release --target x86_64-pc-windows-msvc` (cross or native). If the build fails on any removal, that feature stays in. No partial removals — the goal is a clean build, not maximum elimination.

## 4. Release CI workflow

### 4.1 Trigger

```yaml
on:
  push:
    tags: ['v*.*.*']
```

Fires only on semver-shaped tags. Normal pushes and PRs are unaffected; `ci.yml` continues to handle those.

### 4.2 Job: `release`

`runs-on: windows-latest`. Single job — no matrix.

**Steps:**

1. `actions/checkout@v4`
2. `dtolnay/rust-toolchain@stable` (no extra components needed)
3. `Swatinem/rust-cache@v2`
4. `cargo build --release --target x86_64-pc-windows-msvc`
5. Install UPX via Chocolatey (`choco install upx -y`). `windows-latest` runners have Chocolatey pre-installed.
6. Pack: `upx --best --lzma target/x86_64-pc-windows-msvc/release/alt3rsnap.exe`
7. Compute checksum (PowerShell): `(Get-FileHash ... -Algorithm SHA256).Hash` → write `alt3rsnap-sha256.txt`
8. `softprops/action-gh-release@v2` — creates the GitHub Release from the tag message, attaches `alt3rsnap.exe` and `alt3rsnap-sha256.txt`. `draft: false`, `prerelease: false`.

### 4.3 Permissions

The workflow needs `contents: write` to create releases. Set at the job level:

```yaml
permissions:
  contents: write
```

No `GITHUB_TOKEN` secret configuration required — the default token is sufficient.

### 4.4 UPX note

`--lzma` gives the best compression ratio (typically 50–65% reduction on top of an already-stripped binary). Build time cost: ~5–10 seconds on `windows-latest`. Some AV scanners flag UPX-packed binaries; this is a known trade-off — the user chose UPX with this awareness.
